Glosarium (A – Z)
Semua istilah teknis dan bisnis yang digunakan dalam dokumen ini, diurutkan secara alfabetis.

Istilah
Definisi
ACID
Atomicity, Consistency, Isolation, Durability — empat properti yang menjamin integritas transaksi database. PostgreSQL mendukung ACID penuh.
ADR
Architecture Decision Record — dokumen satu halaman yang mencatat keputusan arsitektur penting beserta alasan, alternatif yang dipertimbangkan, dan konsekuensinya.
API
Application Programming Interface — antarmuka yang memungkinkan dua sistem berkomunikasi. Backend Ngemiloh menyediakan REST API yang dikonsumsi frontend.
Base Price
Harga dasar produk sebelum ditambahkan harga modifier. Diskon dihitung dari base price saja (Opsi A yang sudah dikunci).
Bcrypt
Algoritma hashing satu arah untuk PIN dan password. Cost factor = 12 di production. Tidak bisa di-decrypt — hanya bisa diverifikasi dengan membandingkan hash.
BOM
Bill of Materials — daftar bahan baku beserta takaran yang dibutuhkan untuk membuat satu porsi produk. Digunakan untuk kalkulasi HPP di Fase 2.
Break-even
Titik impas — jumlah revenue minimum yang diperlukan untuk menutup semua biaya tetap. Ngemiloh: Rp 84.500/hari.
BullMQ
Library antrian pekerjaan (job queue) berbasis Redis untuk Node.js. Digunakan untuk async processing: sync offline orders, export CSV, email notifikasi.
Cache
Penyimpanan sementara data yang sering diakses untuk mengurangi beban database. Redis digunakan sebagai cache layer di Ngemiloh.
Caddy
Web server dan reverse proxy modern yang mendukung HTTPS otomatis via Let's Encrypt. Built-in di Coolify, menangani SSL termination.
CI/CD
Continuous Integration / Continuous Deployment — otomatisasi proses test dan deploy via GitHub Actions. Push ke main → test otomatis → deploy otomatis.
Clean Architecture
Pola arsitektur yang memisahkan kode menjadi layer: Domain → Application → Infrastructure → Presentation. Business logic tidak bergantung pada framework.
Coolify
Platform as a Service (PaaS) open-source yang di-deploy di VPS sendiri. Mengelola container Docker, environment variables, dan auto-deployment.
CSRF
Cross-Site Request Forgery — serangan yang memaksa user melakukan aksi yang tidak diinginkan. Dicegah dengan Double Submit Cookie pattern.
Depresiasi
Penyusutan nilai aset dari waktu ke waktu. Formula: Nilai Aset ÷ Umur Pakai (bulan). Total depresiasi Ngemiloh: ~Rp 21.000/bulan.
Docker
Platform containerisasi yang memungkinkan aplikasi berjalan dalam environment yang terisolasi dan konsisten di semua server.
DoD
Definition of Done — kriteria yang harus terpenuhi sebelum sebuah fitur dinyatakan selesai. Mencakup code review, testing, dan dokumentasi.
DoR
Definition of Ready — kriteria yang harus terpenuhi sebelum sebuah fitur boleh mulai dikerjakan oleh developer.
ESC/POS
Escape/Point of Sale — protokol komunikasi standar untuk printer thermal receipt. Digunakan via Web Bluetooth API untuk print struk di Ngemiloh.
F&B
Food and Beverage — industri makanan dan minuman. Ngemiloh adalah bisnis jajanan (snack) F&B.
Feature Flag
Toggle on/off untuk fitur tertentu tanpa perlu redeploy aplikasi. Superadmin bisa mengaktifkan atau menonaktifkan fitur dari dashboard.
FC% (Food Cost%)
Persentase biaya bahan baku terhadap harga jual. Rumus: (HPP ÷ Harga Jual) × 100%. Target Ngemiloh: 30-60% gross margin (FC% 40-70%).
FIFO Costing
First In First Out — metode valuasi stok yang menggunakan harga beli batch pertama untuk menghitung HPP. Batch berikutnya dipakai setelah batch pertama habis.
Foodpail
Kemasan berbentuk kotak untuk produk jajanan. Harga Rp 1.450/pcs (Rp 14.500/ikat isi 10 pcs). Kemasan utama produk Ngemiloh.
Graceful Shutdown
Proses menghentikan server dengan aman: menyelesaikan request yang sedang berjalan, menutup koneksi DB, dan flushing queue sebelum berhenti.
GRN
Goods Receipt Note — dokumen pencatatan penerimaan barang dari supplier. Di Ngemiloh: input manual saat bahan baku dari Ngemiloh Pusat tiba.
Halal MUI
Sertifikasi halal dari Majelis Ulama Indonesia. Ngemiloh sudah memiliki sertifikat ini. Nomor sertifikat akan ditampilkan di struk transaksi.
HPP
Harga Pokok Penjualan — total biaya bahan baku untuk menghasilkan satu unit produk. HPP Macaroni dasar: Rp 5.450/porsi (tanpa saus/bumbu).
Idempotency
Properti operasi di mana menjalankan hal yang sama berkali-kali menghasilkan hasil yang sama seperti satu kali. client_uuid di orders menjamin ini.
IndexedDB
Database NoSQL built-in browser untuk menyimpan data offline. Digunakan di POS kasir untuk antrian transaksi saat koneksi internet terputus.
JWT
JSON Web Token — token terenkripsi yang berisi informasi user (id, role). Disimpan di HttpOnly Cookie. Access token: 8 jam (1 shift kasir).
Kasir
Karyawan/mitra yang mengoperasikan mesin POS. Status: freelance. Bagi hasil: 40% dari laba bersih bulanan. Hanya bisa akses halaman POS.
Laba Bersih
Revenue − HPP Bahan Baku − Biaya Operasional (Opex) − Depresiasi. Ini adalah basis perhitungan bagi hasil Owner (60%) dan Kasir (40%).
MDR
Merchant Discount Rate — biaya yang dikenakan payment gateway per transaksi. QRIS Midtrans: 0% untuk transaksi ≤ Rp 500.000 (merchant mikro).
Middleware
Fungsi yang dieksekusi sebelum request mencapai controller. Di NestJS: AuthGuard dan RolesGuard adalah middleware untuk autentikasi dan otorisasi.
Modifier
Tambahan pilihan pada produk yang mengubah komposisi dan harga. Contoh: pilih bumbu tabur (+ Rp 1.000-2.000) dan pilih saus (+ Rp 1.500-3.500).
NIB
Nomor Induk Berusaha — identitas pelaku usaha dari sistem OSS. Diperlukan untuk pendaftaran merchant Midtrans. Urus di oss.go.id (gratis).
NestJS
Framework Node.js untuk membangun backend API. Menggunakan TypeScript dan Decorator-based Dependency Injection. Struktur modular memudahkan Clean Architecture.
NPWP
Nomor Pokok Wajib Pajak. Ngemiloh sudah memiliki NPWP pribadi atas nama owner. Diperlukan untuk laporan pajak dan pendaftaran merchant.
Opex
Operational Expenses — biaya operasional tetap: sewa, listrik, gas, kemasan, transportasi. Di Ngemiloh: estimasi Rp 1.500.000/bulan, diinput manual owner.
ORM
Object-Relational Mapping — library yang mengabstraksi query SQL menjadi kode TypeScript. Ngemiloh menggunakan Prisma ORM v5.
P&L
Profit and Loss Statement (Laporan Laba Rugi) — laporan keuangan yang menunjukkan revenue, biaya, dan profit dalam satu periode. Dibuat harian di Ngemiloh.
PgBouncer
Connection pooler untuk PostgreSQL. Mengurangi overhead koneksi dengan mendaur ulang koneksi DB yang ada. Dikelola oleh Coolify.
PIN
Personal Identification Number — 4 digit kode rahasia kasir untuk login ke POS. Di-hash dengan Bcrypt cost=12 + pepper sebelum disimpan ke DB.
POS
Point of Sale — sistem transaksi penjualan. Di Ngemiloh: PWA mobile-first yang dijalankan di HP Android kasir (BYOD).
PPh Final UMKM
Pajak Penghasilan Final untuk Usaha Mikro Kecil Menengah. Tarif: 0% untuk omset ≤ Rp 500 juta/tahun (PP 55/2022). Ngemiloh tidak kena PPh Final.
Prisma
ORM modern untuk Node.js dan TypeScript. Menyediakan type-safe database client dan migration system (Prisma Migrate).
PWA
Progressive Web App — aplikasi web yang dapat diinstall di homescreen HP layaknya app native. Mendukung offline mode via Service Worker.
QRIS
Quick Response Code Indonesian Standard — standar pembayaran QR Code Indonesia oleh Bank Indonesia. Diimplementasikan via Midtrans di Ngemiloh.
RBAC
Role-Based Access Control — sistem pembatasan akses berdasarkan peran. Ngemiloh: 2 role (kasir hanya akses POS, superadmin akses penuh).
Redis
In-memory database yang digunakan sebagai cache, job queue (BullMQ), rate limiter, dan token blacklist. Container terpisah di VPS Ngemiloh.
REST
Representational State Transfer — gaya arsitektur API menggunakan HTTP methods (GET, POST, PATCH, DELETE) dan URL yang representatif.
Service Worker
Script JavaScript yang berjalan di background browser, memungkinkan offline functionality, background sync, dan caching untuk PWA.
Settlement
Proses transfer dana dari Midtrans ke rekening bank merchant. Biasanya T+1 atau T+2 dari tanggal transaksi.
SLA
Service Level Agreement — perjanjian tingkat layanan. Uptime target Ngemiloh: 99.5%/bulan (max downtime ~3 jam 40 menit/bulan).
SLO
Service Level Objective — target kualitas layanan yang terukur. Contoh: P95 response time < 500ms untuk POST /orders.
Soft Launch
Strategi membuka bisnis dengan fitur minimal yang berfungsi, kemudian upgrade bertahap. Ngemiloh: buka dengan fitur tunai saja dalam 2 minggu, QRIS menyusul.
Split Payment
Pembayaran yang dibagi antara dua metode: sebagian tunai + sebagian QRIS dalam satu transaksi. Didukung sejak Fase 1 Ngemiloh.
SSE
Server-Sent Events — protokol komunikasi satu arah dari server ke browser. Digunakan untuk push notifikasi pembayaran QRIS sukses ke POS kasir.
Superadmin
Role tertinggi di sistem Ngemiloh = Owner. Akses penuh ke semua fitur, data keuangan, laporan, dan konfigurasi sistem. Login: email + password.
SvelteKit
Framework full-stack berbasis Svelte untuk membangun aplikasi web. Bundle size kecil, ideal untuk HP Android. Menggunakan adapter-static untuk Ngemiloh.
Technical Debt
Shortcut teknis yang diambil untuk kecepatan development, yang harus "dibayar" di masa depan dengan refactoring. Didokumentasikan di Technical Debt Register.
TypeScript
Superset dari JavaScript yang menambahkan static type checking. Digunakan di seluruh stack Ngemiloh (backend + frontend). Mode: strict.
UUID v4
Universally Unique Identifier — string 36 karakter yang unik secara global. Digunakan sebagai primary key di semua tabel Ngemiloh.
Void
Pembatalan transaksi yang sudah berstatus completed. Tidak menghapus data, hanya mengubah status order ke "voided". Hanya bisa dilakukan Superadmin.
VPS
Virtual Private Server — server virtual yang disewa untuk menjalankan aplikasi. Ngemiloh: 4GB RAM, 4 vCPU, 60GB SSD. Provider: IDCloudHost/Vultr.
Webhook
HTTP request yang dikirim secara otomatis oleh sistem eksternal (Midtrans) ke server Ngemiloh ketika ada event tertentu (pembayaran berhasil).
Yield Factor
Faktor penyusutan bahan baku saat proses masak. Di Ngemiloh: yield = 100% (tidak ada penyusutan yang signifikan untuk produk kering).

1. Ringkasan Eksekutif & Konteks Bisnis
Ngemiloh F&B adalah bisnis jajanan (snack) dengan model franchise/mitra. Arif Hidayat adalah mitra pemegang outlet — membeli bahan baku dari Ngemiloh Pusat dan menjalankan operasional dengan 1 kasir freelance.

1.1 Lingkup Sistem
Mesin Kasir PWA (Progressive Web App) — diakses di HP Android kasir (BYOD), Mobile-First, Offline-First.
Dashboard Superadmin — diakses via browser laptop atau HP oleh owner, Full Access ke semua data dan konfigurasi.

💡 Filosofi Sistem
Kasir tidak boleh terhenti karena masalah jaringan. Sistem dirancang untuk selalu menerima transaksi online maupun offline. Setiap keputusan teknis mengutamakan: kecepatan transaksi kasir, keandalan data keuangan, dan kemudahan pengelolaan oleh owner tanpa keahlian IT.

4. Business Rules Master List
Semua business rules berikut sudah FINAL dan dikunci berdasarkan jawaban kuesioner. Tidak dapat diubah tanpa proses Change Request formal.

4.1 Rules Keuangan
ID
Rule
Detail
BR-K01
Formula harga item
final_price = base_price − discount_from_base + sum(modifier_additional_prices). Diskon HANYA dari base price (Opsi A Final).
BR-K02
Formula diskon PERCENTAGE
discount_amount = base_price × (discount_value ÷ 100). Tidak berlaku untuk modifier.
BR-K03
Formula diskon FIXED
discount_amount = min(discount_value, base_price). Tidak boleh melebihi base price.
BR-K04
Formula subtotal
subtotal = final_price × quantity. Disimpan eksplisit di DB untuk laporan cepat.
BR-K05
Formula laba bersih
Laba Bersih = Revenue − HPP Bahan Baku − Biaya Operasional (Opex) − Depresiasi Aset.
BR-K06
Bagi hasil owner
Porsi Owner = Laba Bersih × 60%.
BR-K07
Bagi hasil kasir
Porsi Kasir = Laba Bersih × 40%. (1 kasir = terima seluruh 40%).
BR-K08
Periode bagi hasil
Dihitung: akhir bulan berjalan. Dibayar: tanggal 1–5 bulan berikutnya.
BR-K09
Bagi hasil saat rugi
Jika Laba Bersih ≤ 0: sistem notifikasi ke superadmin. Owner bayar kasir secara kebijaksanaan. Tidak ada angka minimum wajib.
BR-K10
PPh Final UMKM
Tidak berlaku — omset Ngemiloh < Rp 500 juta/tahun (PP 55/2022). Tidak perlu dihitung di sistem.
BR-K11
Depresiasi aset
Metode garis lurus: Depresiasi/bulan = Nilai Aset ÷ Umur Pakai (bulan). Booth = Rp 0 (gratis dari Ngemiloh Pusat).
BR-K12
Valuasi stok
FIFO Costing: harga beli batch pertama digunakan untuk HPP. Saat batch habis, pakai harga batch berikutnya.

4.2 Rules Operasional
ID
Rule
Detail
BR-O01
QRIS hanya saat online
QRIS tidak dapat diproses saat offline. UI wajib tampilkan pesan jelas dan arahkan ke pembayaran tunai.
BR-O02
Split payment
Satu transaksi boleh dibayar sebagian tunai + sebagian QRIS. cash_amount + qris_amount HARUS = total_amount.
BR-O03
Void QRIS refund
Jika transaksi QRIS di-void: refund ke pelanggan dilakukan manual tunai dari kas. Sistem catat refund_method dan refund_at.
BR-O04
Minimum QRIS
Konfirmasi ke Midtrans saat pendaftaran. Default sistem: minimum Rp 1.000. Estimasi bisnis: transaksi > Rp 20.000.
BR-O05
Diskon dua tumpuk
Jika satu produk terkena dua diskon aktif: ambil nilai TERBESAR. Tidak kumulatif.
BR-O06
Stok opname
Dilakukan mingguan oleh owner. Sistem menyediakan fitur input fisik dan kalkulasi selisih otomatis (Fase 2).
BR-O07
Kas awal shift
Kasir input nominal kas awal (default Rp 500.000) saat mulai shift. Kas akhir dihitung saat tutup shift.
BR-O08
Setor kas
Dilakukan setiap 2–3 hari sekali. Sistem mencatat akumulasi kas, bukan hanya harian.
BR-O09
Void transaksi
Hanya superadmin. Wajib isi alasan minimal 10 karakter. Konfirmasi 2 langkah. Alert email jika > 3 void dalam 10 menit.
BR-O10
Produk nonaktif
Tidak tampil di POS. Hard delete DILARANG jika produk sudah pernah ada di order_items.
BR-O11
Harga produk historis
Perubahan harga TIDAK mempengaruhi laporan historis. price_at_time disimpan di order_items.
BR-O12
Retensi data
3 tahun real-time di DB aktif. Setelah 3 tahun: di-export ke file CSV dan di-archive dari DB aktif.

6. App Flow & Peta Navigasi
6.1 Alur Autentikasi
Browser load → cek JWT di HttpOnly, Secure, SameSite=Strict Cookie.
Tidak ada atau expired → redirect ke /login.
Kasir input PIN 4 digit → POST /api/v1/auth/login → backend hash+pepper PIN → bandingkan dengan DB.
Superadmin input email + password → POST /api/v1/auth/login → Bcrypt verify.
Jika valid → server set JWT cookie (8 jam) + CSRF token → redirect ke /pos (kasir) atau /admin/dashboard (superadmin).
Setiap request ke protected endpoint → NestJS AuthGuard verifikasi JWT + RolesGuard verifikasi role.
Token expired → interceptor frontend kirim POST /api/v1/auth/refresh → server rotate refresh token → issue access token baru.

NestJS @Sse() decorator. EventSource di browser. Heartbeat 30 detik anti-disconnect.
Bluetooth Print
Web Bluetooth API + ESC/POS
Native Chrome Android. ESC/POS adalah standar printer thermal.
Support: Epson, GOOJPRT, Xprinter. Fallback: tampil struk di layar.
Job Queue
BullMQ + Redis
Async processing yang fault-tolerant dengan retry mechanism.
Queues: SYNC_OFFLINE, EXPORT_CSV, SEND_EMAIL, UPDATE_DISCOUNT_STATUS, CALC_PROFIT_SHARE.
Email Notifikasi
Nodemailer + Gmail SMTP
Gratis, sudah built-in NestJS ecosystem. Tidak butuh external service.
App password Gmail. Template HTML untuk alert dan reminder.
Monitoring
UptimeRobot + Sentry + Umami
UptimeRobot: uptime alert. Sentry: error tracking. Umami: usage analytics self-hosted.
Semua gratis (Sentry free tier, Umami self-hosted di VPS yang sama).
Infrastruktur
VPS 4GB + Coolify + Docker + Cloudflare
Coolify abstraksi Docker Compose. Container internal: latensi 0ms. Cloudflare: CDN + WAF.
IDCloudHost atau Vultr Tokyo. Ubuntu 24.04 LTS. Caddy auto-SSL. Docker network: ngemiloh_net.

10. Database Schema Final (PostgreSQL 16)
Konvensi Wajib
snake_case untuk semua nama tabel (plural) dan kolom. UUID v4 via gen_random_uuid() sebagai PK. Semua nilai uang: NUMERIC(12,2) — TIDAK BOLEH float/double. Setiap tabel wajib punya created_at TIMESTAMPTZ. Soft delete via deleted_at atau is_active. Audit log: READ-ONLY dari aplikasi.

13. Security Design
13.1 Authentication Flow Detail
User input kredensial → frontend POST /api/v1/auth/login.
Backend rate-limit check: apakah IP sudah ≥5 gagal dalam 10 menit? Jika ya → HTTP 429.
Backend query user dari DB. Verifikasi PIN/password via Bcrypt + pepper.
Jika valid: generate JWT (access 8 jam) + refresh token (30 hari). Set sebagai HttpOnly, Secure, SameSite=Strict Cookie.
Tambah ke audit_log: USER_LOGIN dengan ip_address.
Setiap protected request: AuthGuard extract JWT dari cookie → verify signature → check expiry → extract {id, role}.
RolesGuard check: apakah role user memiliki akses ke endpoint ini?
Access token expired: frontend interceptor otomatis POST /auth/refresh → backend rotate refresh token (hapus lama, buat baru) → set cookie baru.
Logout: DELETE refresh token dari DB → clear cookies.

13.2 PIN Security
📄  PIN Security Implementation
// PIN hashing dengan Bcrypt + pepper
async function hashPin(pin: string): Promise<string> {
  const pepper = process.env.PIN_PEPPER_SECRET;
  // Pepper ditambahkan di kedua sisi untuk keamanan maksimal
  const pepperedPin = `${pepper}:${pin}:${pepper}`;
  return bcrypt.hash(pepperedPin, 12);
}
 
async function verifyPin(pin: string, hash: string): Promise<boolean> {
  const pepper = process.env.PIN_PEPPER_SECRET;
  const pepperedPin = `${pepper}:${pin}:${pepper}`;
  // timingSafeEqual digunakan di dalam bcrypt.compare secara internal
  return bcrypt.compare(pepperedPin, hash);
}
 
// ATURAN PENTING:
// 1. PIN_PEPPER_SECRET TIDAK BOLEH diubah setelah production
//    (semua PIN harus di-reset jika pepper berubah)
// 2. PIN plaintext TIDAK BOLEH ada di log, error message, atau response
// 3. Bcrypt rounds = 12 di production, 10 di development (test lebih cepat)
 

13.3 Webhook Security (Midtrans)
📄  Webhook Security
// Signature Verification — WAJIB, tidak boleh di-skip
// Formula Midtrans: SHA512(order_id + status_code + gross_amount + server_key)
 
function verifyMidtransSignature(payload: any, serverKey: string): boolean {
  const raw = `${payload.order_id}${payload.status_code}${payload.gross_amount}${serverKey}`;
  const expected = crypto.createHash("sha512").update(raw).digest("hex");
 
  // timingSafeEqual mencegah timing attack
  const a = Buffer.from(expected, "hex");
  const b = Buffer.from(payload.signature_key || "", "hex");
  if (a.length !== b.length) return false;
  return crypto.timingSafeEqual(a, b);
}
 
// Webhook handler — SELALU return HTTP 200 ke Midtrans
// Jika return non-200: Midtrans retry berkali-kali → flooding
@Post("qris/webhook")
async handleWebhook(@Body() payload: any) {
  if (!this.verifyMidtransSignature(payload, this.serverKey)) {
    this.logger.warn(`Invalid signature: order ${payload.order_id}`);
    return { status: "IGNORED" }; // Tetap 200!
  }
  await this.paymentQueue.add("process-webhook", payload);
  return { status: "OK" }; // Return segera, proses di background
}

17. Architecture Decision Records (ADR)
ADR mendokumentasikan keputusan arsitektur penting beserta konteks dan alasannya. Ini mencegah keputusan yang sama dipertanyakan berulang kali.

ADR-001: Formula Diskon — Opsi A (Base Price Only)
Status
ACCEPTED
Konteks
Sistem memiliki dua mekanisme yang mempengaruhi harga: modifier (tambah harga) dan diskon (kurangi harga). Perlu dikunci bagaimana keduanya berinteraksi.
Keputusan
Diskon dihitung HANYA dari base_price. Modifier tidak terpengaruh diskon. Formula: final_price = (base_price − discount) + sum(modifiers).
Alasan
Lebih mudah dikomunikasikan ke pelanggan ("diskon pada harga produk, bukan pada pilihan topping"). Lebih mudah diimplementasikan dan ditest. Keputusan bisnis owner (konfirmasi tertulis tersedia).
Alternatif
Opsi B: Diskon dari total termasuk modifier. Lebih menguntungkan pelanggan tapi lebih kompleks dan ambigu secara bisnis.
Konsekuensi
Semua unit test kalkulasi harga harus menggunakan formula ini. Tidak dapat diubah tanpa migrasi data dan update seluruh test suite.

ADR-002: SvelteKit adapter-static (bukan adapter-node)
Status
ACCEPTED
Konteks
SvelteKit bisa dijalankan sebagai SSR (server-side rendering dengan Node.js container) atau sebagai file statis (build-time rendering).
Keputusan
Gunakan adapter-static: build menjadi file HTML/CSS/JS statis yang di-serve langsung oleh Caddy.
Alasan
Sistem POS adalah aplikasi internal yang tidak butuh SEO. Keamanan dijamin di API level (NestJS), bukan di level SvelteKit server. Hemat ~256MB RAM VPS (tidak perlu Node.js container terpisah untuk frontend).
Alternatif
adapter-node: mendukung SSR, route protection server-side, tapi butuh container tambahan dan RAM lebih.
Konsekuensi
Route protection hanya bisa dilakukan di client-side (cek JWT di browser). Semua security enforcement tetap di NestJS API — ini sudah cukup dan lebih tepat.

ADR-003: Redis untuk Cache, Queue, dan Rate Limit
Status
ACCEPTED
Konteks
Beberapa komponen membutuhkan state yang persisten di luar PostgreSQL: cache produk, job queue async, rate limit counter, token blacklist.
Keputusan
Gunakan Redis 7 sebagai single service untuk semua kebutuhan tersebut. BullMQ untuk job queue, @nestjs/cache-manager untuk cache, @nestjs/throttler Redis store untuk rate limit.
Alasan
Redis adalah standar industri untuk use case ini. Satu service menangani semua kebutuhan. Lebih murah dan sederhana dari mengelola beberapa service terpisah.
Alternatif
In-memory NestJS: lebih sederhana tapi tidak persisten saat server restart dan tidak bisa di-share antar instance. PostgreSQL untuk queue: bisa tapi performa lebih buruk.
Konsekuensi
Menambah satu container di Docker Compose. Total RAM +128MB. Redis harus di-backup (AOF).

ADR-004: NestJS + TypeScript (bukan Go)
Status
ACCEPTED
Konteks
Backend bisa dibangun dengan berbagai bahasa. Go menawarkan performa lebih tinggi. Node.js/NestJS lebih familiar dan ekosistemnya lebih kaya untuk proyek ini.
Keputusan
Gunakan NestJS v10 dengan TypeScript strict mode.
Alasan
Satu bahasa (TypeScript) di seluruh stack mengurangi context-switching untuk solo developer. Ekosistem NestJS kaya (Prisma, BullMQ, Nodemailer semua first-class). Performa Node.js cukup untuk skala 1 outlet.
Alternatif
Go: lebih cepat, lebih hemat resource, tapi ekosistem lebih terbatas dan learning curve lebih tinggi untuk proyek baru.
Konsekuensi
Jika ada bottleneck performa di Fase 3 (multi-outlet, 100+ transaksi/menit), evaluasi migrasi ke Go untuk service kritis. Flag: ADR-007 (PENDING).

ADR-005: PostgreSQL Range Partitioning per Tahun
Status
ACCEPTED
Konteks
Tabel orders akan bertumbuh. Dalam 3 tahun dengan 100 transaksi/hari: ~109.500 orders. Query laporan tanpa partitioning akan makin lambat.
Keputusan
Partisi tabel orders berdasarkan client_created_at per tahun: orders_2026, orders_2027, orders_2028.
Alasan
Query "laporan bulan ini" hanya scan 1 partisi, bukan seluruh tabel. Archiving lebih mudah: DETACH PARTITION orders_2026 setelah 3 tahun. Vacuum lebih efisien per partisi.
Alternatif
No partitioning: lebih sederhana tapi query makin lambat seiring data bertumbuh. Partisi per bulan: terlalu granular untuk skala ini.
Konsekuensi
Schema lebih kompleks. Prisma tidak support partitioning natively — perlu raw SQL untuk create partition. Partition baru harus dibuat sebelum tahun baru (via cron job).

ADR-006: Modifier System dengan product_modifier_groups + options
Status
ACCEPTED
Konteks
Produk Ngemiloh memiliki banyak varian berdasarkan pilihan bumbu + saus dengan harga tambahan berbeda-beda per opsi.
Keputusan
Arsitektur dua tabel: product_modifier_groups (grup pilihan) + product_modifier_options (opsi per grup dengan additional_price). Order mencatat pilihan di order_item_modifiers.
Alasan
Fleksibel: satu produk bisa punya banyak grup, satu grup bisa punya banyak opsi. Snapshot tersimpan di order_item_modifiers sehingga perubahan harga modifier tidak mempengaruhi laporan historis.
Alternatif
Single variant table (satu baris per kombinasi produk+bumbu+saus): lebih sederhana tapi tidak scalable jika bumbu/saus bertambah. Hard-coded modifier di products: tidak fleksibel.
Konsekuensi
Menambah 2 tabel baru dan 1 junction table. Logika kalkulasi harga lebih kompleks. Unit test wajib untuk semua kombinasi.
